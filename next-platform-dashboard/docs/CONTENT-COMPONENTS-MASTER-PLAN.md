# DRAMAC CMS — Content Components Master Plan

## Executive Vision

Transform DRAMAC's 8 content components from a functional but inconsistently registered set into a **cohesive, theme-aware, accessibility-first content rendering library** capable of producing Medium-quality rich text, Notion-grade code blocks, and editorial-quality quotes — all generated by the AI Designer with **zero human adjustment**.

Content components are the **substance of every website**. They carry the message — the words, stats, quotes, and code that visitors read, scan, and act on. If content components are polished — beautiful typography, seamless dark mode, accessible semantics, and full AI control — every site the platform generates reads and feels professional. This plan treats the 8 content components as the foundational text layer beneath every section, card, and page DRAMAC produces.

## 3. Architecture Principles

### 3.1 Inline Styles for All Colours (No Hardcoded Tailwind)

```typescript
// ❌ WRONG — Hardcoded Tailwind colour class
"text-gray-500"

// ✅ CORRECT — Inline style with CSS variable fallback
style={{ color: color || 'var(--color-muted-foreground, inherit)' }}
```

**Rule:** Every user-facing colour must be applied via `style={{}}` with CSS variable fallbacks. Tailwind classes are **only for structural** properties (padding, flex, grid, transitions, opacity).

**Exception:** CodeBlock theme colours are intentional per-theme presets. They should migrate to CSS variables but the current approach is acceptable since the themes are self-contained colour systems.

### 3.2 CSS Variables as Typography Tokens

All content components follow a consistent pattern for font inheritance:

```typescript
// Font family — inherits from brand settings when empty
fontFamily: fontFamily || `var(--font-body, inherit)`    // Body text
fontFamily: fontFamily || `var(--font-heading, inherit)`  // Headings/display

// Font size — uses design token scale
fontSize: fontSize || `var(--type-sm, 0.875rem)`          // Size from token
fontSize: fontSize || `var(--type-5xl, 3rem)`              // Large display

// Line height — uses token scale
lineHeight: `var(--leading-${titleLevel}, 1.2)`            // Heading level

// Letter spacing — uses token scale
letterSpacing: `var(--tracking-${titleLevel}, -0.02em)`    // Heading level
```

**Used by:** RichText (heading), Quote (body), Label (body), List (body), DisplayText (heading), DividerText (body), StatNumber (heading for value, body for label).

### 3.3 Colour Resolution Chain

Content components use a cascading colour resolution pattern:

```typescript
// RichText example — most complex colour chain:
const resolvedTextColor = textColor || color || undefined;          // Step 1: explicit → fallback → inherit
const resolvedTitleColor = titleColor || accentColor || resolvedTextColor;  // Step 2: specific → accent → text
const resolvedPullQuoteColor = pullQuoteColor || accentColor || resolvedTitleColor;  // Step 3: deep chain
```

**Rule:** The final fallback is always `undefined`, which lets CSS variables and inherited styles take over. **Never hardcode** a colour as the final fallback.

### 3.4 Render Props Match Registry Field Names Exactly

```
converter.ts normalizes AI output →  stored as { type, props }  →  core-components.ts fields validate/display
                                                                      ↓
                                                           renders.tsx receives via {...injectedProps}
```

**Critical:** Render function parameter names MUST match registry field `name` values exactly. There is NO intermediate mapping layer. If the registry has `text` and the render expects `content`, the prop is silently lost.

### 3.5 Semantic HTML by Default

| Component | Required Semantic Element | Current | Status |
|-----------|-------------------------|---------|--------|
| RichText | `<section>` + `<hN>` + `<blockquote>` | ✅ All present | ✅ |

#### Shared Utility: markdownToHtml()

Defined between the interface (L3575) and the export (L3694), this utility converts markdown syntax to HTML. It handles:
- Headings (`#` through `######`)
- Bold (`**text**`), Italic (`*text*`)
- Inline code (`` `code` ``)
- Links (`[text](url)`)
- Unordered lists (`- item`)
- Ordered lists (`1. item`)
- Paragraphs (double newline)

**Shared with:** Accordion and Tabs components (not just content category).

## 5. Shared Utility: markdownToHtml()

### 5.1 Location

Defined in `renders.tsx` between L3575 (after `RichTextProps` interface) and L3694 (before `RichTextRender` export). This is a **module-level function**, not component-specific.

### 5.2 What It Converts

| Markdown | HTML Output |
|----------|-------------|
| `# Heading` → `######` | `<h1>` → `<h6>` |
| `**bold**` | `<strong>` |
| `*italic*` | `<em>` |
| `` `code` `` | `<code>` |
| `[text](url)` | `<a href="url">text</a>` |
| `- item` | `<ul><li>` |
| `1. item` | `<ol><li>` |
| Double newline | `<p>` break |

### 5.3 Consumers

| Component | Category | Usage |
|-----------|----------|-------|
| RichText | Content | Main body `content` prop |
| Accordion | Interactive | Item content rendering |
| Tabs | Interactive | Tab content rendering |

### 5.4 Security Model

`markdownToHtml()` is an allowlist-based parser — it only converts recognised patterns to HTML. It does **not** pass through arbitrary HTML from user input. The output is used in `dangerouslySetInnerHTML`, which is safe because:
1. The parser only produces known-safe HTML tags (`<h1>`-`<h6>`, `<strong>`, `<em>`, `<code>`, `<a>`, `<ul>`, `<ol>`, `<li>`, `<p>`)
2. Link URLs are not sanitised (CMS content is trusted) — but if user-facing input is ever supported, URL sanitisation should be added

### 5.5 Limitations

- No table support
- No image support
- No horizontal rule (`---`) support
- No nested lists
- No task lists (`- [x] done`)
- No footnotes
- No front matter

These are acceptable for a CMS — the `content` field is a richtext editor, not a documentation system. markdown-to-HTML is a convenience for AI-generated content.

### 6.2 RichText Prose Dark Mode Fix

The `prose` class from Tailwind CSS applies default colours for headings, paragraphs, links, and code. In dark mode, these default colours are dark-on-light and become invisible on dark backgrounds.

**Current code:**
```tsx
<div className={`${proseSizeClass} max-w-none prose-headings:font-bold prose-p:leading-relaxed ...`}
     style={{ color: resolvedTextColor || undefined }}>
```

**Fix:**
```tsx
<div className={`${proseSizeClass} max-w-none
     prose-headings:font-bold prose-p:leading-relaxed
     [&_*]:!text-inherit`}  // Force all prose children to inherit colour
     style={{ color: resolvedTextColor || undefined }}>
```

Or alternatively, apply `color: inherit` to all prose child elements via the inline style approach, ensuring the `resolvedTextColor` cascades to all child elements. The existing `--tw-prose-links` override for `accentColor` shows this approach is already partially used.

### 6.3 CodeBlock Theme Selection in Dark Mode

CodeBlock uses self-contained themes — the `dark` and `github` themes work naturally on dark backgrounds, while `light` is designed for light backgrounds. The AI Designer should select the appropriate theme:

```typescript
// AI rule: CodeBlock theme should match page background
```

### 6.4 CSS Variable Inheritance Pattern

All 7 non-CodeBlock components follow the same pattern:

```typescript
// Colour prop → explicit inline style (highest priority)
// undefined   → CSS variable fallback (from brand/theme tokens)
// undefined   → inherit from parent (lowest priority)

style={{ color: textColor || undefined }}
// If textColor is "#ff0000" → uses #ff0000
// If textColor is undefined → no inline style → inherits from parent/CSS variable
```

This ensures zero-effort dark mode — if no explicit colour is set, components inherit from the page's CSS variables which change between light/dark themes.

| `--font-body` | System sans-serif | RichText body, Quote, Label, List, DividerText, StatNumber label |

### 8.2 Type Scale Tokens

| Token | Default | Used By |
|-------|---------|---------|
| `--type-sm` | `0.875rem` | StatNumber label, Label sizing |
| `--type-base` | `1rem` | List base font size |

| `--type-5xl` | `3rem` | StatNumber value |
| `--type-9xl` | `8rem` | DisplayText default |

### 8.3 Leading (Line Height) Tokens

| Token | Used By |
|-------|---------|
| `--leading-h1` through `--leading-h4` | RichText title elements |

### 8.4 Tracking (Letter Spacing) Tokens

| Token | Used By |
|-------|---------|
| `--tracking-h1` through `--tracking-h4` | RichText title elements |

### 8.5 How Tokens Flow

```
1. typography-intelligence.ts → analyses brand → selects font pairings
2. color-intelligence.ts → extracts primary/accent/background colours
3. renderer.tsx → injects CSS variables into iframe <style>:
     --font-heading: 'Montserrat', sans-serif;
     --font-body: 'Open Sans', sans-serif;
     --type-5xl: 3rem;
4. Content components read variables as defaults:
     fontFamily: fontFamily || `var(--font-heading, inherit)`
5. Any explicit prop override (set in registry UI) takes precedence
```

### 9.3 AI Quality Scoring for Content

```typescript
function scoreContentQuality(page: GeneratedPage): { score: number; issues: string[] } {
  let score = 100;
  const issues: string[] = [];

  // Rule: Blog/about pages should use RichText, not raw HTML
  if (page.type === "blog" && !page.hasComponent("RichText")) {
    score -= 15;
    issues.push("Blog page should use RichText component for body content");
  }

  // Rule: Stats section should use StatNumber, not plain text
  page.sections.forEach(section => {
    if (section.title?.toLowerCase().includes("stat") && !section.hasComponent("StatNumber")) {
      score -= 10;
      issues.push(`Section "${section.title}" discusses statistics but doesn't use StatNumber component`);
    }
  });

  // Rule: Feature lists should use List with check/arrow, not bullet
  page.getComponents("List").forEach(list => {
    if (list.props.variant === "bullet" && page.isFeatureSection(list.parentSection)) {
      score -= 5;
      issues.push("Feature list should use 'check' or 'arrow' variant, not 'bullet'");
    }
  });

  // Rule: CodeBlock on non-tech pages is unusual
  if (page.industry !== "technology" && page.hasComponent("CodeBlock")) {
    score -= 5;
    issues.push("CodeBlock on non-tech page — verify it's intentional");
  }

  // Rule: DisplayText should only appear in hero/splash contexts
  page.getComponents("DisplayText").forEach(dt => {
    if (!page.isHeroSection(dt.parentSection)) {
      score -= 5;
      issues.push("DisplayText used outside hero section — consider using Heading instead");
    }
  });

  return { score: Math.max(0, score), issues };
}
```

3. **Registry fields must match render prop names exactly** — no intermediate mapping, props pass through via `{...injectedProps}`
4. **Every component needs a converter normalizer** — handles AI alias props and establishes sensible defaults

### Specific Component Rules
5. **RichText** — `prose` class needs dark mode handling; `markdownToHtml()` is allowlist-based (safe for `dangerouslySetInnerHTML`)

7. **CodeBlock** — MUST register all 10 props; MUST add converter normalizer; theme colours are self-contained (hardcoded by design)
8. **Label** — `overline` variant is `display: block` — the only variant that alters layout flow
9. **List** — `numbered` → `<ol>`, all others → `<ul>` — semantic HTML required
10. **DisplayText** — `role="heading" aria-level={1}` — always uses ARIA heading role, not `<h1>` tag

12. **StatNumber** — `fontVariantNumeric: "tabular-nums"` — never remove this; MUST add composite `aria-label`
