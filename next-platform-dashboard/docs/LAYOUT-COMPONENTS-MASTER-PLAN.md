# DRAMAC CMS — Layout Components Master Plan

## Executive Vision

Transform DRAMAC's layout component system from a basic set of 6 primitive containers into a **world-class, award-winning layout engine** capable of producing websites that rival Awwwards winners, Webflow showcases, and Framer portfolio sites — all generated by the AI Designer with **zero human adjustment**.

Layout components are the **foundation of every page**. Every other component — text, images, forms, modules — lives inside a layout. If layouts are perfect, everything placed inside them will be perfect. This plan treats layouts as the most critical category in the entire component system.

## 1. Current State Audit

### Existing Layout Components (6 total)

| Component | Location | Lines | Quality | Key Issues |
|-----------|----------|-------|---------|------------|
| **Section** | `renders.tsx` SectionRender | ~80 | ⚠️ Basic | Parallax prop exists but NOT implemented. No scroll-snap. No sticky. No min/max height control. No gradient backgrounds. No video backgrounds. |
| **Container** | `renders.tsx` ContainerRender | ~60 | ⚠️ Basic | Only `maxWidth` sizing. No aspect-ratio. No container queries. No overflow control beyond hidden. No backdrop filters. |
| **Columns** | `renders.tsx` ColumnsRender | ~50 | 🔴 Broken | Uses `grid-cols-${cols}` template literals — **fails Tailwind's build-time class scanning**. No unequal column widths. No auto-fit/fill. No named grid areas. No subgrid. |
| **Card** | `renders.tsx` CardRender | ~60 | ⚠️ Basic | Decent hover effects but no glassmorphism, no gradient borders, no animated borders. Limited to flat/shadow styles. |
| **Spacer** | `renders.tsx` SpacerRender | ~15 | 🔴 Broken | Only supports `number` heights — **ignores ResponsiveValue objects**. No visual divider mode. Fixed pixel only. |
| **Divider** | `renders.tsx` DividerRender | ~50 | ✅ Decent | Color, thickness, style, width fraction, optional text. Decent but could use gradient support and animated variants. |

### Critical Architecture Issues

1. **No block files for 4/6 components** — Columns, Card, Spacer, Divider only exist as renders. No editor field definitions, no toolbar controls.
2. **Responsive class maps are incomplete** — `paddingYMap`, `paddingXMap`, `gapMap`, `borderRadiusMap` exist but cover limited value ranges. No responsive width, height, margin, or font-size maps.
3. **Premium components are MORE advanced** — `premium-components.tsx` has Flexbox, Grid, and Container layout primitives with far more features than the core layout components. This is backwards.
4. **Universal props under-integrated** — `render-wrapper.tsx` applies animation/hover/visibility but uses CSS class-based animations, not Framer Motion. Limited to predefined presets.
5. **AI Designer doesn't leverage layout potential** — Prompts in `prompts.ts` mention layout types but provide minimal guidance on when to use advanced layout features.
6. **No z-index management** — Components can stack visually but there's no layering system.
7. **No position control** — No sticky, fixed, absolute, or relative positioning options.

### What Premium Components Do Better

The existing `premium-components.tsx` already has:
- **2D/3D parallax** on Hero sections (but not available as a reusable layout prop)
- **Scroll indicators** (but hardcoded to Hero, not reusable)
- **Video backgrounds** (Hero only)
- **Advanced navbar with scroll detection** (120+ props)
- **Dynamic multi-column footers**
- **Full flexbox/grid layout primitives** (registered as premium but not in core)

These prove the platform CAN do advanced layouts — they're just locked inside premium components instead of being core layout capabilities.

## 3. Architecture Principles

### 3.1 The Layout Promise
Every layout component MUST guarantee:

1. **Any component works inside it** — Drop a form, a carousel, a video, a map, ANYTHING inside any layout and it renders perfectly.
2. **Responsive by default** — Every dimension, spacing, and visual property adapts across mobile/tablet/desktop with ZERO manual adjustment.
3. **Animatable** — Every layout can entrance-animate, scroll-animate, hover-animate, and transition between states.
4. **Nestable infinitely** — Sections contain Containers contain Columns contain Cards and it all works.
5. **AI-generatable** — The AI Designer can use every layout feature without ambiguity.
6. **Performance-first** — Layouts use CSS for static properties, Framer Motion only for dynamic animations. No layout shifts. No CLS.

### 3.2 Responsive Value System (Enhanced)

```typescript
// Current (limited)
type ResponsiveValue<T> = { mobile: T; tablet?: T; desktop?: T };

// Enhanced
type ResponsiveValue<T> = {
  mobile: T;
  tablet?: T;
  desktop?: T;
  // Future: landscape?: T; ultrawide?: T;
};

// Every layout property that varies by breakpoint uses this
type ResponsiveSize = ResponsiveValue<string>; // "0" | "2" | "4" | "8" | "12" | "16" | "20" | "24" | "32" | "48" | "64" | "auto"
type ResponsiveDisplay = ResponsiveValue<"block" | "flex" | "grid" | "none">;
type ResponsivePosition = ResponsiveValue<"static" | "relative" | "sticky" | "fixed">;
```

### 3.3 Tailwind-Safe Class Generation

**CRITICAL FIX:** Replace ALL template literal class generation with lookup maps.

```typescript
// BROKEN (current Columns approach)
className={`grid-cols-${cols}`} // Tailwind CANNOT scan this

// CORRECT (map-based)
const gridColsMap: Record<number, string> = {
  1: "grid-cols-1", 2: "grid-cols-2", 3: "grid-cols-3",
  4: "grid-cols-4", 5: "grid-cols-5", 6: "grid-cols-6",
  7: "grid-cols-7", 8: "grid-cols-8", 9: "grid-cols-9",
  10: "grid-cols-10", 11: "grid-cols-11", 12: "grid-cols-12",
};
```

Every responsive class must follow this pattern. No exceptions.

### 3.4 Style Injection Strategy

Properties fall into two categories:

| Category | Method | Examples |
|----------|--------|---------|
| **Static** | Tailwind classes via maps | padding, margin, gap, display, position, overflow, border-radius |
| **Dynamic** | Inline styles | custom colors, gradients, transforms, clip-path, backdrop-filter, exact dimensions |

Framer Motion's `motion.div` is used ONLY when:
- Element needs entrance/exit animations
- Element needs scroll-driven transforms
- Element needs layout animations (shared layout)
- Element needs gesture-based interactions (drag, hover physics)

#### Section Render Implementation Strategy

```tsx
// Section becomes a motion.section (conditionally)
const SectionRender = ({ props, children, brandColors }) => {
  const needsMotion = props.scrollReveal || props.parallaxSpeed;

  const Wrapper = needsMotion ? motion.section : "section";

  // Parallax: Uses Framer Motion useScroll + useTransform
  // scrollReveal: Uses Framer Motion useInView + animate
  // Video bg: <video> element behind content with absolute positioning
  // Shape dividers: Inline SVG positioned absolute top/bottom
  // Gradient: Computed CSS string applied as backgroundImage
  // Multiple layers: CSS multiple backgrounds with correct ordering

  return (
    <Wrapper {...motionProps}>
      {shapeDividerTop && <ShapeDivider position="top" {...shapeDividerTop} />}
      {backgroundVideo && <VideoBackground {...backgroundVideo} />}
      {overlay && <Overlay {...overlay} />}
      <div className={innerContainerClasses}>
        {children}
      </div>
      {shapeDividerBottom && <ShapeDivider position="bottom" {...shapeDividerBottom} />}
    </Wrapper>
  );
};
```

```typescript
function resolveResponsive<T extends string>(
  value: ResponsiveValue<T> | T | undefined,
  maps: { mobile: Record<T, string>; tablet: Record<T, string>; desktop: Record<T, string> }
): string {
  if (!value) return "";
  if (typeof value === "string") return maps.mobile[value] || "";
  const classes: string[] = [];
  if (value.mobile) classes.push(maps.mobile[value.mobile] || "");
  if (value.tablet) classes.push(maps.tablet[value.tablet] || "");
  if (value.desktop) classes.push(maps.desktop[value.desktop] || "");
  return classes.filter(Boolean).join(" ");
}
```

### 7.3 Mobile-First Cascade Rules

1. **Mobile styles are always set.** No breakpoint prefix. These are the default.
2. **Tablet overrides only what changes.** Uses `md:` or `sm:` prefix.
3. **Desktop overrides only what changes.** Uses `lg:` prefix.
4. **If a responsive value only has `mobile` set**, that value applies everywhere (inheritance).
5. **If tablet is undefined, it inherits from mobile.** This is CSS's natural cascade.

### 7.4 AI Designer Responsive Rules

The AI Designer MUST be instructed to:
- Always provide mobile values (they're required in `ResponsiveValue`)
- Only provide tablet/desktop overrides when the value should CHANGE
- Use larger paddings on desktop, smaller on mobile
- Use more columns on desktop, fewer on mobile
- Stack horizontal layouts vertically on mobile
- Increase font sizes on desktop (handled by text components)
- Never use fixed pixel widths that would overflow on mobile

## 14. Mobile-First Perfection Strategy

### 14.1 Mobile Layout Rules (Non-Negotiable)

1. **No horizontal overflow.** Every layout component must `max-width: 100%` on mobile. No side scrolling unless explicitly intended (carousel/scroll mode).
2. **Touch-friendly spacing.** Minimum 44px tap targets. Minimum 8px gap between interactive elements.
3. **Stack by default.** All multi-column layouts render as single-column on mobile unless explicitly overridden.
4. **Readable text.** Containers must have minimum `px-4` (16px) side padding on mobile.
5. **No tiny text.** Font sizes never below 14px on mobile (handled by text components but layouts shouldn't squeeze text containers).
6. **Images contained.** All images within layout components must be `max-width: 100%` and `height: auto`.
7. **Videos responsive.** Video backgrounds pause on mobile (battery), show poster image instead.
8. **Animations simplified.** Scroll-driven animations disabled on mobile. Entrance animations kept but simplified (shorter duration, less travel distance).
9. **Safe areas.** Account for notches, dynamic toolbars, and rounded corners via `padding: env(safe-area-inset-*)`.

### 14.2 Mobile Testing Checklist

Every layout component render must pass:

- [ ] Renders correctly at 320px width (iPhone SE)
- [ ] Renders correctly at 375px width (iPhone 12/13/14)
- [ ] Renders correctly at 390px width (iPhone 14 Pro)
- [ ] Renders correctly at 412px width (Pixel 7)
- [ ] No horizontal scrollbar at any width
- [ ] Text is readable without zooming
- [ ] Interactive elements are tappable
- [ ] Safe area insets respected on notched devices
- [ ] No CLS (Cumulative Layout Shift) during load
- [ ] No FOUC (Flash of Unstyled Content)

### 14.3 App-Like Mobile Experience

To make sites "look splendid on mobile and fit perfectly like mobile apps":

1. **View Transitions API** — Smooth page transitions (similar to native app navigation)
2. **Overscroll behavior** — `overscroll-behavior: none` on scroll-snap sections (prevents rubber-banding)
3. **Touch action control** — Layout components set appropriate `touch-action` (e.g., `pan-y` for vertical scroll sections)
4. **Momentum scrolling** — `-webkit-overflow-scrolling: touch` on scrollable containers
5. **Pull-to-refresh prevention** — On full-page layouts, prevent default pull-to-refresh
6. **Scroll restoration** — Maintain scroll position on back navigation
